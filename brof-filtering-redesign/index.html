BRoF Filtering Redesign

Business Review of the Future (BRoF) is a business analytics web application that doubles as a presentation tool for senior-level managers at Colgate-Palmolive.

A month after working on the general user interface of the application*, I was brought back to improve the app's filtering interaction.

* You can find my work with the general layout of BRoF here.

Role
UX Design

  • Approach

    FIltering is an essential aspect of practically every analytics tool, and BRoF is no exception.

    I decided upon a chronological approach to this task. I drafted a simple filtering user flow and solved the problems as and when they crop up while a user attempts to filter their dashboard.

  • Problems & Solution

  • Visual clutter

    There can be up to three filtering groups on any one screen: geography, product, and category. Initially, all groups are displayed on the menu bar. If the user wants to filter their dashboard, they are immediately forced to think of which group their filter should fall under; this introduces a mental burden very early on in the process.

    Not only that, generating and populating the filtering modal takes some time. Should a user click on the wrong group, they then have to exit, click, and load the modal for the correct group. This makes for a very costly mistake.

    The solution was to group them into a tabbed row in a single filtering modal summonable via the filter icon on the right. This delays the mental burden by a click, and requires only one filtering modal to be generated and populated.

  • Singular column introduces memory load

    The original design utilized a singular column. When an item is selected, for example 'Asia Pacific' here, it is moved to the top of the list, and everything else is italicized. This is to indicate that it is not related to the current selected items.

    It was a conscious design choice by the developers to implement only one list, citing user feedback for a simple interface, however this introduces a major issue. When the user selects an item, it is moved to the top without any indication. When they are further down on a long list, it seems to disappear abruptly from their perspective.

    Even should users familiarize themselves with this response, it requires that the user remembers what they have already selected. Also, selecting an item by mistake incurs a substantial cost. To unselect, the user would have to scroll back up to locate the selected sublist.

    Hence, I proposed a two column interface. One mainly for selecting and providing a quick way to undo a selection, and the other to remind users of their current selection and to allow for easy removal of any item.

  • Removing filters from the dashboard

    Developers implemented a strip at the top of the dashboard that displays the current filters and affords users the ability to remove filters with a click. However, there was a huge drawback — removing a filter item forces a reload of the entire dashboard.

    Unfamiliar users would be swayed by the interface to think that this strip is the easiest way to remove existing filter items. However, attempting to remove more than one item from this strip will cause multiple reloads. Unless users are particularly interested in conducting iterative analyses by removing a single filter item at a time, this does more harm than good.

    Not only that, when more than a handful of filters are applied, the list truncates with an ellipsis. This means that unless the filter item is near the top of the filter list, users would still have to remove it through the appropriate filter modal.

    As an alternative, I proposed that the selection list be displayed in a nested structure in the new filtering modal. This is so that a group of items could be removed simultaneously. For example, deleting the 'Asia Pacific' filter item also deletes 'South East Asia' from the filter. This also reduces clutter on the dashboard.
